# GitHub Pages 上线操作清单

适用场景：本项目使用 React + Vite，已配置 CI、部署工作流与 e2e 检查。

## 1. 仓库前置确认

- [ ] 本地安装依赖后可正常构建。
  - 执行：`npm ci`
  - 执行：`npm run build`
- [ ] 发布前校验通过。
  - 执行：`npm run verify:release`
- [ ] 默认分支已确认（通常为 `main`）。
- [ ] 仓库已包含 GitHub Actions 部署工作流文件（位于 `.github/workflows/`）。

## 2. GitHub 设置（Pages + Actions 权限）

- [ ] 进入仓库 `Settings -> Pages`。
- [ ] `Build and deployment` 的 `Source` 选择 `GitHub Actions`。
- [ ] 进入 `Settings -> Actions -> General`。
- [ ] `Workflow permissions` 选择 `Read and write permissions`。
- [ ] 勾选允许工作流创建和批准 Pull Request（若页面有该选项）。
- [ ] 保存设置。

## 3. 首次手动触发部署步骤

- [ ] 进入仓库 `Actions` 页面，找到部署工作流。
- [ ] 点击 `Run workflow`。
- [ ] 分支选择默认分支（通常 `main`），点击运行。
- [ ] 等待工作流完成，确认关键 Job 全部成功。
- [ ] 首次发布成功后，记录线上地址。
  - GitHub Pages URL 规则：`https://<github-username>.github.io/<repo-name>/`

## 4. 线上验证项

- [ ] 打开线上 URL，确认首页可访问且无 404。
- [ ] 强制刷新浏览器缓存后再次访问，确认静态资源正常加载。
- [ ] 检查关键页面路由可访问。
- [ ] 检查控制台无明显报错（尤其是资源路径和脚本加载错误）。
- [ ] 按发布前的关键业务路径进行一次冒烟验证。

## 5. 失败排查顺序

- [ ] 先看 `Actions` 失败步骤日志，定位失败命令。
- [ ] 回到本地复现失败命令。
  - 常用：`npm ci`
  - 常用：`npm run verify:release`
  - 常用：`npm run build`
- [ ] 若构建成功但页面异常，检查 `base` 路径配置是否与仓库名一致。
- [ ] 检查 Pages 设置是否仍为 `Source: GitHub Actions`。
- [ ] 检查工作流权限是否被改回只读。
- [ ] 修复后重新触发部署，再做线上验证。

## 6. 日常发布建议

- [ ] 每次发布前固定执行：`npm run verify:release`。
- [ ] 合并到默认分支后观察 Actions 运行结果，确保自动部署成功。
- [ ] 发布后用无痕窗口做一次最小冒烟验证。
- [ ] 若有版本说明，记录发布时间、提交范围、线上 URL。
- [ ] 遇到失败按“失败排查顺序”逐项执行，不跳步。
